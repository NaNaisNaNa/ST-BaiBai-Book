# 带数据创建新对话(Carryover)— 暂定方案

> 临时文档,待第二个新功能确定后一起改代码。参考 SillyTavern-Horae `createNewChatWithCarryover`(index.js:19073)。

## 目标形态

旧对话(一大堆历史楼层 + 最近 N 条全文窗口)→ 创建新对话,带过去:

```
#0 锚点楼(隐藏 is_system=true,正文空/仅占位)
   ├─ extra.bbs_leaf  = 「种子叶子」(L0):text=合并历史摘要;delta=当前全量状态
   └─ (森林里另置一条 L2 总结节点,childIds 收纳该种子叶子)   ← 见「层级」决定
#1.. 保留窗口的旧楼层(原样全文 + 各自 extra.bbs_leaf 一起搬过来)
```

新对话靠现成重放管线(deriveMemory:空白起步逐叶子 fold)自动还原状态,注入(setExtensionPrompt)自然就对,**无需特殊种子载体**——这是柏宝书相对 Horae 的根本简化(Horae 要拷一整份 meta,柏宝书只要造一片把状态编码成全量 add 的叶子)。

## 关键决策(已和用户确认)

1. **合并历史摘要 = 直接拼**,不再调副 API 重压(对齐 Horae:拼现有摘要文本,省一次请求/省 token)。
   - 来源:窗口起点 `carryStart` 之前的剧情,用 `selectHistoryNodesBefore(memory.summaries, chat, carryStart)` + `renderHistoryNodes(...)` 选「最高存活压缩层」节点并按时序拼接(现成函数,inject.ts)。
2. **合并后算 L2 总结**(用户明确:对应 Horae 的 `level:'摘要'` = 柏宝书 L2)。
   - ⚠️ 约束冲突:柏宝书「压缩节点(MemSummary,level≥1)只压文本、不带 delta、不参与重放」。而「携带当前状态」必须靠一片带 delta 的**叶子**来重放。
   - 解法:**种子叶子(L0,带全量状态 delta,挂 #0 的 extra)承载状态重放;另造一条 L2 MemSummary 承载合并摘要文本,childIds 收纳这片种子叶子**。
     - 但 L1 缺位:森林约定 L2.childIds 指向 L1,L1.childIds 指向叶子。直接 L2→叶子会跳级。
     - 两个可选实现,二选一(写代码前定):
       - **A. 跳级直挂**:L2.childIds 直接放种子叶子 id。需确认 buildView/selectInjectionNodes/pruneBrokenComps 对「L2 直接含叶子」不报错(看代码:childIds 只按 id 查 byId,不强制层级连续,**应可行**;collectFloors/collectLeaves 是递归到 leaf 即停,也 OK)。
       - **B. 补齐 L1**:种子叶子 ← L1 ← L2,造两层壳。更贴合约定但更重。
     - **倾向 A**(更简单,代码未假设层级连续)。写代码时实测 pruneBrokenComps 不误删。
   - 时间:L2 节点 timeStart/timeEnd 取合并范围首尾;种子叶子 timeEnd 取旧对话当前 state.time(做时间锚)。
3. **保留窗口楼层不可能已被压缩**(用户确认),搬运时只搬叶子(随消息 extra 自然跟随),压缩节点不搬,新对话森林只放那条 L2。
4. **保留条数**沿用 `apiSettings.keepRecent`(默认 5,数 AI 消息条数),复用 `resolveKeepStart(chat)` 求窗口起点 `carryStart`。

## 种子叶子的 delta(全量状态编码)

以旧对话**当前派生状态**(`deriveMemory(chat)` 全量,或 `memory.items/plans/state` 现成镜像)编码:

```ts
delta = {
  time: memory.state.time || undefined,
  location: memory.state.location || undefined,
  items: { add: memory.items.map(i => ({
    name: i.name, qty: i.qty, desc: i.desc,
    carried: i.carried, location: i.location,   // 随身/寄存原样带过
  })) },
  plans: { add: memory.plans.filter(p => p.status==='open').map(p => ({
    kind: p.kind, content: p.content,
    createdTime: p.createdTime, targetTime: p.targetTime,
  })) },
}
```

- 全量 add 重放即还原(apply.ts 带符号累加,从空白起,一次 add 即得目标量)。
- 已 resolved 的 plan 不带(新对话不需要历史已结计划)。
- itemLog 是派生产物,不用带(新对话从种子叶子重放会自然生成「获得 X」日志;可接受,或后续优化)。

   - `seedDelta` = 上面的全量状态(截止 carryStart 之前?还是全量?**需定**:Horae 用「截止 carryStart 的累积状态」。柏宝书应取 `deriveMemory(sourceChat, carryStart)` —— 即窗口之前的状态;窗口楼层自己的叶子搬过去会继续累加,避免重复计入)。

# 第二个新功能:向量记忆(Vector Memory)— 暂定方案

> 参考 Horae `core/vectorManager.js`(已通读全 4197 行)。后端柏宝库在 `D:\1\SillyTavern\plugins\baibaoku`(也是本人插件,可改)。

## 用户已定的三个大方向

1. **检索下沉到柏宝库后端**(实现细节自选)。← 解决「计算慢 + 跨聊天」的核心。
2. **角色卡级自动绑定**:同一角色卡(用 avatar 唯一键)的所有聊天共享一个向量库;带数据建新聊天后自动召回该角色旧剧情。零配置。
3. **只索引叶子摘要**:每条叶子的 `text`(单楼摘要)各做一条向量。索引数 = 叶子数。与现有数据模型最贴合。

## Horae 现状与两个问题的根源(为什么必须一起解)

- Horae 架构:embedding(API/本地)→ 前端内存 `Map<msgIndex, vector>` → 持久化两级(IndexedDB 本地一级缓存 + 柏宝库跨设备同步)→ 检索时前端 `for` 循环逐条 `_dotProduct` 暴力扫描 → 阈值/RRF/rerank。
- **柏宝库在 Horae 里只是「向量仓库」**:存 `cacheKey(=文档hash+模型hash,内容寻址) → float32 向量`,不带归属、不做检索、不算相似度。
- **问题1(慢)**:`search()` O(N) 暴力点积,跑前端主线程。
- **问题2(只能本聊天)**:`this.vectors` 由 `loadChat(chatId, chat)` 从**当前 chat 的消息**构建(`_collectVectorTasks` 遍历当前聊天)。柏宝库的缓存是纯内容寻址、无「属于哪个记忆库」的归属 → 新聊天 chat 里没有旧楼层 → 检索范围里就没旧内容。
- **耦合**:解决问题2后,向量规模从「单聊天几百」→「角色全历史几万」,问题1的前端 O(N) 彻底撑不住。故两者必须一起做 = **存储+检索整体下沉后端**。

## 柏宝库后端现状(已确认可扩展)

- `plugins/baibaoku`:纯 KV + SQLite(better-sqlite3 / node:sqlite),serializer 已支持 `float32` 类型。
- `DatabaseManager`(src/database.js):干净的 KV 抽象,`kv` 表(store,key,value,type,...)。
- `api.js`:`router.post('/v1/<action>', ...)` 清晰模式,加新接口很自然。
- 数据落 `data/<user>/baibaoku/databases/<database>.sqlite`,随用户目录备份。

## 召回范围 = 哈希包列表(最终方案,推翻谱系推断)

⚠️ **推翻早先两版**(角色卡级 memory_id、chat_lineage 谱系推断)。最终用**内容寻址哈希包**,把「谁和谁有关系」整个消灭——不再让前/后端推断 parent。

**核心机制**:
1. **带数据建新聊天时**,前端把源聊天的「全部楼层正文 + 对应摘要 + 时间(+ 向量)」打包成一个对象,POST 给后端;后端存下、返回一个 **bundle 哈希**(内容寻址,不可变快照)。
2. 前端拿到哈希**才正式建新聊天**,把哈希写进新聊天 `chat_metadata`(一个**哈希列表**字段)。
3. **哈希列表会累加**:已含 `[hash_A]` 的 B 建出 C → 打包 B **自己**的内容得 `hash_B` → C 的列表 = B 的列表 + 新哈希 = `[hash_A, hash_B]`。(只打包源的"自己内容",A 已由 hash_A 覆盖,不重复打包。)
4. 以后每次向量操作(索引/检索),前端**把哈希列表一起发后端** → 后端把这些 bundle 的向量纳入可召回范围。
```
A ──功能一──> B(metadata.bundles=[hash_A])
              B ──功能一──> C(metadata.bundles=[hash_A, hash_B])
A 召回: A 自己
B 召回: B 自己 + bundle(A)
C 召回: C 自己 + bundle(A) + bundle(B)
```
全新聊天 D:metadata 无 bundles → 只召回自己。

**为什么这版根治一切**:
- **零关系推断**:可召回范围 = 聊天数据里明写的哈希列表,后端不建谱系表、不上溯。
- **分支天然解决**:哈希列表在 `chat_metadata` 里,ST 建原生分支**物理拷贝 metadata** → 分支自动继承同一份哈希列表,**无需任何分支检测代码**。分支自己物理含多少楼层无所谓(用户提醒:分支从某条消息建,不一定含源全部内容)——它索引自己实际有的内容即可。
- **快照语义正确**:bundle 是建子那刻的冻结快照,C 召回的是 B 当时的内容,B 后续改动不进 C(符合"从某时刻分叉")。
- **抗删除**:bundle 内容寻址独立存,源聊天被删,bundle 仍在,召回不断链。

## 后端方案(柏宝库升级:向量仓库 → 向量数据库)

新增一组 `/v1/vec/*` 接口(与现有 KV 接口并列,不动老接口)。两类「可召回集合」,用统一 `scope` 键区分:
- `scope = chat:{chatId}` —— 某聊天**自己**实时索引的叶子(边玩边长)。
- `scope = bundle:{hash}` —— 建子时冻结的快照包(不可变)。

一张向量表即可:
```sql
CREATE TABLE vec_items (
  scope      TEXT NOT NULL,   -- 'chat:<chatId>' 或 'bundle:<hash>'
  leaf_id    TEXT NOT NULL,   -- 来源叶子 id(去重/更新键)
  doc_hash   TEXT NOT NULL,   -- 文档内容 hash(内容寻址,embedding 复用)
  vector     BLOB NOT NULL,   -- float32
  dim        INTEGER NOT NULL,
  document   TEXT NOT NULL,   -- 叶子摘要正文(召回回传 / 摘要档)
  mes_full   TEXT,            -- 楼层原文(召回全文档用;跨聊天召回必须存)
  story_time TEXT,            -- 故事内时间(展示/相对时间)
  msg_index  INTEGER,         -- 楼层索引(展示/溯源)
  created_at INTEGER,
  PRIMARY KEY (scope, leaf_id)
);
```
**向量始终存后端,前端不持有向量库**(纠正早先表述):前端只负责「调 embedding API 把文本变向量」这个动作,拿到向量**立刻 upsert 给后端存**,自己不留;检索只把 query 向量发后端,后端拿它跟库里的向量算。

- **接口**:
  - `vec/bundle/create`(sourceChatId) → **后端内部把 `chat:<sourceChatId>` 的全部行复制成 `bundle:<hash>` scope**(`INSERT ... SELECT ... WHERE scope='chat:<源>'`),hash 对该集合内容算,**返回 hash**。向量本就在后端,**不经前端、不重 embed**。幂等:同内容同 hash,重复 create 直接返回已存 hash。
  - `vec/upsert`(scope=`chat:<chatId>`, items[])— 当前聊天实时索引(leaf_id 冲突即覆盖)。items=`[{leaf_id, doc_hash, vector, document, mes_full, story_time, msg_index}]`。**这是唯一前端上传向量的入口**(刚 embed 出来的叶子向量)。
  - `vec/search`(scopes[], queryVectors[], rerankCandidates, excludeLeafIds)— **scopes = `['chat:<当前>', 'bundle:<hash_A>', 'bundle:<hash_B>'…]`(前端直接给全集合,后端不推断)**。`WHERE scope IN (scopes)` 限定范围,算相似度,纯按得分取前 N(不套阈值)返回 `{leaf_id, scope, similarity, document, mes_full, story_time, msg_index}`。前端只发 query 向量,不发库向量。
  - `vec/delete`(scope, leaf_ids[])— 删当前聊天叶子时同步删(只删 `chat:` scope;bundle 不可变不删)。
  - `vec/stats`(scopes[])— 各 scope 索引数。
- **检索实现二选一**(写代码时定,倾向先纯 JS,留 sqlite-vec 升级位):
  - **首选稳妥**:纯 JS 点积(`SELECT ... WHERE scope IN (...)` 全取 → Node 算)。不阻塞前端、Node 更快、零依赖。
  - **进阶**:`sqlite-vec`(`vec0`)原生 KNN,optionalDependency 缺失回退纯 JS。预留 dim 字段与接口形态。
- **embedding 这个动作在前端**(API key/CORS 在前端):前端把文本 embed 成向量,随即 upsert 给后端存。检索只把 query 向量发后端。**向量库本身在后端,前端不持有**。
- 内容寻址:doc_hash 做 embedding 复用(同文本不重复 embed)。bundle **不需要前端 embed**——后端内部从源聊天 `chat:` scope 复制现成向量。

## 前端方案(柏宝书侧新增 src/memory/vector*.ts)

- **聊天 metadata 新增字段** `bbs_bundles: string[]`(哈希列表)。随 chat 文件走 → ST 原生分支拷贝 metadata 时自动继承。
- **建包/写哈希(功能一那刻)**:`vec/bundle/create(sourceChatId)`(后端内部复制源向量,前端只传 chatId)→ 得 `hash` → 新聊天 metadata `bbs_bundles = [...源的 bbs_bundles, hash]`。**先建包拿哈希、再正式建聊天写入**(用户强调的时序)。
- **索引时机**:叶子生成/更新后(runSummary 写叶子、syncItemLogFromMessage、editLeafAt 后)→ 前端 embed 该叶子 text → `vec/upsert(scope='chat:<当前chatId>', items=[{leaf_id, doc_hash, vector, document:叶子text, mes_full:楼层原文, story_time, msg_index}])`。删叶子(deleteLeafAt/pruneBrokenComps)→ `vec/delete`。
  - 批量补建:摘要页「重建向量索引」按钮,遍历当前所有有效叶子 embed+upsert(对齐 Horae batchIndex)。
- **召回时机 = 阻塞式**(用户已定,不做预取/异步):生成主回复前(拦截器阶段,与摘要补漏同一处 await),阻塞走完召回再放行。管线:
  1. 构造查询重写上下文(见下「查询重写上下文构造」)。
  2. **Query Rewrite**(借鉴 Horae,可选/可关):小模型把上下文重写成多条检索 query;渠道走向量专用 channel(queryRewrite 角色,留空复用 embedding 渠道)。关掉则直接用「状态快照+用户输入」当单条 query。
  3. 各 query 前端 embed → 发后端 `vec/search(scopes=['chat:<当前>', ...bbs_bundles.map(h=>'bundle:'+h)], queryVectors, rerankCandidates, excludeLeafIds)` → 后端在这些 scope 范围内纯按 embedding 得分取前 `rerankCandidates` 条(**不套 embedding 阈值**)。
  4. 对候选做 rerank(向量 rerank 角色渠道)→ 按下方「召回分档」规则分全文档/摘要档,合计 ≤ finalRecallCount。
  5. **按 leaf_id 去重**(分支拷贝/bundle 快照会让同 id 跨 scope 出现多份,留一条,优先离当前最近的 `chat:`);再排除当前窗口内已全文/已注入的叶子(按 leaf_id exclude)。
  6. 拼成「记忆召回」注入文本,setExtensionPrompt(新槽位,贴近 history 注入)。全文档发 mes_full,摘要档发 document。
- **与带数据建新聊天的协同**:建包即把源聊天叶子(后端内部)冻结进 bundle;新聊天靠 metadata 哈希列表把这些 bundle 纳入召回 scopes。种子叶子(text=合并摘要)随新聊天 `chat:` scope 索引。

### 召回分档规则(最终定稿,对齐 settings.ts VectorRecallSettings)

五个参数(已落 UI:`vector.recall`):rerankCandidates / embeddingThreshold / rerankThreshold / fullTextCount / finalRecallCount。

1. **取候选**:召回 scopes(当前聊天 + 各 bundle)内全部向量算 embedding 相似度,**纯按得分排序取前 rerankCandidates**——不套 embeddingThreshold,哪怕前 N 全是 0.4/0.3 低分也照进(阈值只在第 3 步摘要档用)。
2. **rerank**:候选送 rerank 打分。
3. **分两档**:
   - **全文档** = rerank 得分 ≥ rerankThreshold,取前 fullTextCount 条,发**原文全文**(mes_full)。
   - **摘要档** = rerank 得分 < rerankThreshold **但** embedding 得分 ≥ embeddingThreshold,发**叶子摘要**(document)。
4. **上限**:finalRecallCount 是总条数上限。先放全文档,不足用摘要档补;补不满也无妨(只是上限,不是配额)。

### 查询重写上下文构造(状态快照的精确放置)

整体结构对齐 Horae:**历史剧情摘要 + 最近窗口全文 + 状态快照 + 用户输入**。差别在状态快照怎么取、放哪——这受我们「比 Horae 更松的不变式」影响。

**关键前提:状态只能走 `deriveMemory`,绝不能假设 `getLatestState` 干净可取。**
- Horae 强制「只有最末 AI 可无摘要」;柏宝书在生成时也有同款拦截(`holesExceptLast` + 生成拦截器),但我们**更松**:① 保留窗口的楼层有叶子却发全文(信息比注入摘要更全);② 不变式可被打破(自动摘要关、摘要失败留洞、刚载入未补摘 → 多条 AI 楼无叶子)。
- 因此状态一律由 `deriveMemory(chat, upToExclusive)` 重放现存叶子算出:无叶子的楼自然不贡献状态,但其全文对模型可见。

**状态快照的放置规则(精确版)**:
- 从「最近全文窗口」起点向后扫,在**第一个「无有效叶子」的楼**停下;
- 快照内容 = `deriveMemory(chat, 那个洞楼的索引)`(= 重放该洞之前所有叶子,含早已滚出窗口的物品/计划);
- 快照**插在「停下点前一楼」之后、洞楼之前**。即放在「连续叶子前缀的末尾」之后,后续无叶子楼裸全文跟随。
- 例:窗口 a b c d e,abc 有叶子、d 没有 →
  ```
  [历史剧情摘要]
  a (全文) / b (全文) / c (全文)
  [状态快照 = deriveMemory(chat, d.index)]   ← 截到 c,放此处
  d (全文) / e (全文)                          ← 无叶子,快照管不到
  [用户输入]
  ```
- 正常情况(不变式成立):只有最末 e 无叶子 → 快照截到 d、放 d 后,e 裸全文(与 Horae「放在最后一个可靠状态楼之后」等价)。
- 为何不放到最末(e 之后):快照号称「当前状态」却因 d/e 无叶子漏掉其变化,会与模型刚读到的 d/e 全文自相矛盾。放在 c 后最诚实:「截止 c 的确凿状态在此,d/e 全文自己读增量」。
- 同源印证:主对话注入侧 `inject.ts` 的 `resolveStateInjectionDepth` 已是同一哲学的简化版(最末 AI 有叶子→D1、无→D2),只处理「最末一条」;查询重写是平铺消息序列,可按上述规则精确插。

**状态快照喂什么(精简,避免与全文冗余)**:
- 最近窗口发全文 → 当前时间/地点/在场角色/氛围这些**模型已能从全文读到**,快照里**不必再给**(冗余且稀释 query 焦点)。
- 只保留**已滚出窗口、但长期有效**的:**物品清单 + 未了结计划/悬念**(如「用之前那把匕首」——匕首 30 楼前拿的,最近全文没有,靠 items 帮重写模型把它解析成检索 query)。
- 取舍是「噪声/质量」而非性能(`deriveMemory` 本就在跑)。**v1 可先完全不给快照**,只喂全文+用户输入看召回质量,不够再加 items/plans。

## 两个功能的交汇点

带数据建新聊天(功能一)+ 向量记忆(功能二)= 完整的「长期记忆」闭环:
- 功能一把「最近窗口 + 合并摘要 + 状态」物理搬进新聊天(主上下文可见)。
- 功能二把源聊天打包成 **bundle 快照**留柏宝库,新聊天按需向量召回(主上下文之外、相关才捞)。
- **缝合点 = 哈希列表**:功能一建子那刻打包源→得 hash→写进新聊天 `metadata.bbs_bundles`(累加);功能二召回时把这些 bundle 纳入 scopes。
- 传递:A→B→C,C 的 `bbs_bundles=[hash_A, hash_B]`,逐次累加,天然实现「只召回自己 + 直系来源的快照,不召回旁系/全新聊天」。ST 分支拷贝 metadata → 自动继承,无需特判。
